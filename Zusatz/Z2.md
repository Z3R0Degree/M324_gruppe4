# Z2 – Ticketsystem & Continuous Improvement

Dokumentation – Evaluation und Einbindung eines Ticketsystems (Jira)

### 1. Einleitung

Im DevOps-Umfeld ist Continuous Improvement ein zentraler Bestandteil. Neben der technischen Umsetzung von CI/CD-Pipelines ist es essenziell, Rückmeldungen aus Betrieb und Kundenkontakt strukturiert zu erfassen und in den Entwicklungsprozess zurückzuführen.
Ein Ticketsystem stellt hierbei die Verbindung zwischen Kunden, Support, Entwicklung und Betrieb dar und dient gleichzeitig als Datenquelle für Kennzahlen.

Ziel dieser Arbeit ist die Evaluation und Einbindung eines Ticketsystems in einen DevOps-Gesamtprozess. Als Tool wird **Jira** verwendet.

### 2. Zielsetzung

_Das Ticketsystem soll:_

- Kunden die einfache Erfassung von Tickets ermöglichen

- durch den 1st Level Support bewirtschaftet werden

- eine Eskalation an den 2nd Level Support ermöglichen

- ohne Neuerfassung von Tickets auskommen

- in den Entwicklungs- und CI/CD-Prozess integriert sein

- Commits, Branches und Deployments nachvollziehbar verknüpfen

- Kennzahlen für Continuous Improvement liefern

### 3. Tool-Evaluation: Jira

Jira ist ein etabliertes Issue- und Service-Management-Tool, das sich besonders für DevOps-Prozesse eignet.

**Vorteile:**

- Konfigurierbare Workflows (Support & Entwicklung)

- Service Portal für Kunden

- Rollen- und Berechtigungskonzept

- Native Integration mit Git-Repositories

- Automatische Verknüpfung von Tickets, Commits und Pull Requests

- Umfangreiche Reporting- und Exportfunktionen


### 4. Rollen & Verantwortlichkeiten

**Kunde**

- Erfasst Tickets über ein Service Portal

- Sieht Status und Rückmeldungen

**1st Level Support**

- Erste Analyse und Klassifikation

- Lösung einfacher Anfragen

- Eskalation komplexer Tickets

**2nd Level Support (DevOps / Entwicklung)**

- Technische Analyse

- Umsetzung von Fixes oder Changes

- Einbindung in CI/CD-Pipeline

- Deployment

### 5. Ticket-Workflow
**5.1 Ticket-Erfassung**

- Ticket wird durch den Kunden im Service Portal erstellt

- Ticket erhält automatisch eine eindeutige Ticket-ID (z. B. Z2-123)

****5.2 Bearbeitung durch 1st Level Support**

- Status: Open → In Progress

- Analyse, Rückfragen, einfache Lösungen

- Bei Bedarf Eskalation per Statuswechsel

**5.3 Eskalation an 2nd Level Support**

- Ticket bleibt unverändert bestehen

- Status: Escalated → 2nd Level

- Zuweisung an Entwickler

→ Es wird kein neues Ticket erstellt.

### 6. Einbindung in Entwicklung & CI/CD
**6.1 Branching-Strategie**

Für jedes eskalierte Ticket wird ein eigener Branch erstellt:

- bugfix/Z2-123-fix-login

- feature/Z2-456-add-validation

Die Ticket-ID ist fixer Bestandteil des Branchnamens.

##### 6.2 Commit-Verknüpfung

Commit-Nachrichten enthalten die Ticket-ID:

_Z2-123 Fix login validation error_


Jira erkennt diese Referenz automatisch und verlinkt:

- Commits

- Branches

- Pull Requests

##### 6.3 Code Review & Pipeline

- Pull Request wird erstellt

Automatische CI-Schritte:

- Build

- Tests

- Linting

Code Review (4-Augen-Prinzip)

Merge & Deployment

Der Ticketstatus wird automatisch aktualisiert (z. B. In Review, Done).

### 7. Kennzahlen & Continuous Improvement

Aus Jira können folgende KPIs gewonnen werden:

- Durchlaufzeit eines Tickets

- Anzahl Eskalationen

- SLA-Einhaltung

- Reopen-Rate

- Tickets pro Kategorie

- Time-to-Fix


## Bewertung anhand der Kriterien

| Kriterium                         | Umsetzung                     |
|----------------------------------|-------------------------------|
| Keine Neuerfassung von Tickets   | Status- & Zuweisungswechsel   |
| Eskalation an 2nd Level          | Integrierter Workflow         |
| Branching-Integration            | Ticket-ID im Branch           |
| Commit-Verknüpfung               | Commit-Message-Standard       |
| CI/CD-Einbindung                 | Pipeline & Reviews            |
| KPI-Extraktion                   | Dashboards & API              |

### 9. Fazit

Jira erfüllt alle Anforderungen an ein integriertes Ticketsystem im DevOps-Umfeld.
Es ermöglicht eine durchgängige Verbindung von Kundenanfragen über Support und Entwicklung bis hin zu Deployment und Reporting und stellt damit eine solide Grundlage für Continuous Improvement dar.

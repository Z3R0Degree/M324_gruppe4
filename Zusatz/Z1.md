# Z1: Kennzahlen / Continuous Improvement

## 1. Auswahl der Kennzahlen
Für dieses Projekt wurden folgende Kennzahlen definiert:

* Test Coverage
* Build-Erfolgsrate
* Build-Dauer
* Anzahl Bugs / Issues

Diese Kennzahlen wurden gewählt, da sie direkt mit Qualität, Stabilität und Effizienz der Applikation zusammenhängen.

## 2. Übersicht der Kennzahlen

| Kennzahl          | Quelle                | Beschreibung                                    |
| ----------------- | --------------------- | ----------------------------------------------- |
| Test Coverage     | CI Pipeline           | Anteil des Codes, der durch Tests abgedeckt ist |
| Build-Erfolgsrate | GitHub Actions        | Anteil erfolgreicher Builds                     |
| Build-Dauer       | CI Pipeline           | Zeit, die ein CI-Durchlauf benötigt             |
| Anzahl Bugs       | Ticketsystem / Issues | Anzahl gemeldeter Fehler                        |

## 3. Beschreibung und Interpretation der Kennzahlen

### 3.1 Test Coverage

* **Was wird gemessen:**
  Prozentsatz des Codes, der durch Unit- und Integrationstests abgedeckt ist.
* **Warum wichtig:**
  Eine hohe Testabdeckung reduziert das Risiko von Fehlern im Produktivbetrieb.
* **Verbesserungsmöglichkeit:**
  Zusätzliche Unit Tests für kritische Logik und Randfälle implementieren.

### 3.2 Build-Erfolgsrate

* **Was wird gemessen:**
  Verhältnis zwischen erfolgreichen und fehlgeschlagenen CI Builds.
* **Warum wichtig:**
  Eine hohe Erfolgsrate zeigt stabile Builds und sauberen Code.
* **Verbesserungsmöglichkeit:**
  Frühzeitiges Testen lokal und strengere Code-Reviews.

### 3.3 Build-Dauer

* **Was wird gemessen:**
  Zeit, die die CI Pipeline für einen kompletten Durchlauf benötigt.
* **Warum wichtig:**
  Kurze Build-Zeiten ermöglichen schnelleres Feedback für Entwickler.
* **Verbesserungsmöglichkeit:**
  Caching von Dependencies oder Parallelisierung von Jobs.

### 3.4 Anzahl Bugs / Issues

* **Was wird gemessen:**
  Anzahl gemeldeter Fehler pro Feature oder Release.
* **Warum wichtig:**
  Gibt Aufschluss über die Qualität der implementierten Features.
* **Verbesserungsmöglichkeit:**
  Verbesserte Tests und frühere Fehlererkennung in der CI Pipeline.


## 4. Verwendung der Kennzahlen im Business-Kontext

Die gesammelten Kennzahlen können in Geschäfts- oder Projektberichten verwendet werden, um:

* den aktuellen Qualitätsstand der Applikation aufzuzeigen
* Risiken frühzeitig zu erkennen
* Verbesserungsmassnahmen zu priorisieren
* den Fortschritt über mehrere Iterationen zu vergleichen

Die Kennzahlen sind verständlich dokumentiert, sodass auch nicht-technische Personen (z. B. Projektleitung) den Zustand des Projekts nachvollziehen können.
